Layer 2 — Footnote anchoring (the most underrated problem)
Protocol tables are littered with superscript markers (a, b, c, *, †) that conditionally modify specific cells. A plain LLM call never associates these with the cells they modify — it either ignores them or applies them globally. The solution is a dedicated pre-processing step:
Build a footnote resolution agent that, before the main extraction LLM ever sees the table, detects all superscript markers in the table, matches each marker to its definition in the footnote block, and re-annotates each affected cell in the structured representation with that footnote's content as cell-level metadata. The LLM then sees something like {visit: Screening, procedure: ECG, required: "yes", footnote: "only if QTc > 450ms at baseline"} rather than ECG [a] plus a separately floating a. only if QTc > 450ms at baseline.
A useful analogy from adjacent fields: this is essentially the same problem as reference resolution in legal contracts or citation linking in scientific papers — fields with more mature tooling for this kind of span-level annotation.

Layer 3 — Schema-constrained extraction and grounding
Research recommends addressing LLM hallucination in table extraction through: enhanced layout-aware parsing, calibrating hallucination risk via confidence-aware decoding, and improving structural grounding via schema-constrained output filtering or multi-pass extraction. Nature
In practice this means:
Defining a typed JSON schema for your site budget output before you prompt. The LLM must fill slots in a known schema (visit name, procedure, frequency, cost code, conditional flag, footnote) rather than freestyle generate. Use function calling / structured outputs mode (JSON mode in the API) to enforce this.
Multi-pass extraction with consistency checking: run the extraction twice independently with slightly different prompts, then diff the two outputs. Disagreements flag cells for human review rather than silently picking one answer. This adds cost but dramatically reduces silent hallucination.
Abstention as a first-class output: explicitly prompt the LLM to output a confidence: low field and a raw_source_text field for any cell it is uncertain about, rather than fabricating a plausible value. A human reviewer handles low-confidence cells; the agentic system handles high-confidence ones.

The failure mode of uniform tiling
If you cut a table into a fixed 8-piece grid, your cuts will slice through cells, merged regions, and footnote anchors at arbitrary points. The sub-images lose context — a cell in the bottom-left tile has no idea what its column header is (which is in the top-left tile). The right version: semantically-aware decomposition
Instead of uniform spatial cuts, decompose along the table's own structure. This is actually a multi-resolution analysis pattern — and your instinct about breaking apart and comparing is exactly right, just needs to be structure-led rather than geometry-led.
Here's the architecture that makes this work:
Pass 1 — Whole-table gestalt read. Send the full table image to the VLM at reduced resolution. The goal here isn't to extract values — it's to understand the table's logic: how many visit columns, what the row groupings are, where the merged cells are, what the footnote symbol set is. Output is a structural schema, not data.
Pass 2 — Semantic decomposition into natural sub-regions. Using the structural schema from Pass 1, divide the table into semantically coherent chunks: the header block (column labels + visit names), each row group (e.g. "Safety Assessments", "PK Samples"), and the footnote block. Each chunk is a natural unit — it doesn't cut across merged cells or sever a footnote from its anchor. These chunks do overlap deliberately at their boundaries — each sub-image includes one row of context from the adjacent group.
Pass 3 — High-resolution extraction per chunk. Each semantic chunk gets a focused extraction pass at full resolution, with its context baked into the prompt: "You are reading the Safety Assessments row group. The column headers from Pass 1 are: [Screening, Day 1, Week 4...]. Extract each cell value with its row and column coordinates."
Pass 4 — Reconciliation. Reassemble all chunk extractions into a unified table and run three comparison checks: (a) boundary consistency — do cells at the edges of adjacent chunks agree? The honest problem with classic GANs here
Traditional GANs (Generator + Discriminator) are generative models — they learn to produce realistic outputs. What you're solving is a verification problem: you have a fixed source document and you want to know whether your extraction is faithful to it. A GAN isn't naturally suited to that because there's no generation task — you're not trying to synthesise a table, you're trying to read one accurately.
If you applied a vanilla GAN approach, the discriminator would learn to distinguish "real protocol table extractions" from "fake ones" — but it would be learning from your training data's distribution, not from the specific source document in front of it. That's not the same as verifying faithfulness to a particular table.
Where the adversarial principle is genuinely useful — reframed
The adversarial instinct is right, but the right framing isn't GAN — it's adversarial validation using two agents in opposition. This is a distinct pattern that's much better suited to your problem:
An Extractor agent tries to produce the most complete, faithful extraction it can. An Challenger agent is specifically prompted to find errors — it takes the extracted JSON and the source image and asks: "What has the extractor missed, misread, or fabricated? What cells are inconsistent with what I see in the image?" The Challenger is not trying to generate a better extraction — it's trying to falsify the Extractor's output.
This is much closer to how adversarial robustness works in practice, and it maps well onto your multi-pass pipeline.
Where a trained discriminator does add value
There is one specific place where a learned discriminator in the GAN sense earns its keep: detecting plausible-sounding hallucinations. The hardest failures in your pipeline are not obviously wrong values — they're values that look structurally correct (right format, right range, right type) but are fabricated. A discriminator trained on a corpus of verified protocol extractions could learn the statistical fingerprint of real protocol tables — typical visit frequencies, typical procedure co-occurrence patterns, typical cost ranges — and flag extractions that violate those learned priors even when they look superficially correct.
Think of it less as a GAN and more as an anomaly detector trained on verified ground truth.
A more tractable research direction: consistency-based self-adversarial testing
There's a cleaner version of this idea that doesn't require training a GAN at all, which is relevant given you're likely working with a limited corpus of verified protocol tables. It borrows from the metamorphic testing literature in software engineering:
If your extraction is correct, then certain transformations of the input should produce predictably transformed outputs. For example — if you rotate the table image 90 degrees, reparse, then rotate the output back, the values should be identical. If you crop to a single row group and extract just that group, the values should match what the full-table extraction returned for that group. If you present the table in a different visual rendering (e.g. regenerate it as HTML and screenshot it), the extraction should be identical.

The golden set is only as good as its ground truth, and creating ground truth for complex protocol tables is not trivial. A few principles:
Two independent annotators should produce the ground truth for each document, with a third adjudicating disagreements. The disagreements themselves are valuable — they tell you which cells are genuinely ambiguous to humans, which sets your accuracy ceiling. If two expert annotators disagree on a cell, you cannot reasonably expect your pipeline to get it right consistently either, and that cell should probably always route to human review regardless of pipeline confidence.
Ground truth should be stored at cell granularity in a structured format, not as a "correct output document." You want to be able to run automated comparison at the field level — not just check whether the final document looks right. This means each annotated cell has coordinates, value, data type, applicable footnotes, and a difficulty rating.
The difficulty annotation is the part most teams skip and most regret
Tagging every cell as trivial / moderate / hard / known-failure-type lets you report accuracy broken down by difficulty tier rather than as a single aggregate number. A pipeline reporting 94% overall accuracy sounds fine — until you realise it's 100% on trivial cells and 60% on hard ones, and the hard cells are exactly the ones that drive site budget errors. Difficulty-stratified metrics tell you where to invest next.
Version control the golden set alongside the pipeline
This sounds obvious but is frequently ignored. Every time you update a prompt, change a model, add a pre-processing step, or modify your schema — re-run the full golden set and store the results with a timestamp and pipeline version tag. Over time this builds a performance history that lets you detect regressions immediately and also demonstrate to your client that accuracy is monotonically improving.
